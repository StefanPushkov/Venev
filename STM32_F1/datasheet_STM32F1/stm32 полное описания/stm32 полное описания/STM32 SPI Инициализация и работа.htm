STM32 SPI Инициализация и работа

Чтобы настроить модуль SPI для работы в режиме ведущего (этот режим чаще всего используется)  необходимо выполнить следующие действия:

  1. Задать скорость обмена разрядами BR[2:0] регистра SPI_CR1.
  2. Задать полярность и фазу тактового сигнала разрядами CPOL и CPHA регистра SPI_CR1.
  3. Задать формат данных (8 или 16 бит) разрядом DFF регистра SPI_CR1.
  4. Задать порядок передачи данных (первым или последним битом вперед) разрядом LSBFIRST регистра SPI_CR1.
  5. Если вывод NSS будет использоваться для аппаратной подачи сигнала NSS – установить разряд SSOE регистра SPI_CR1. Если он сконфигурирован как вход этого сигнала – подать на него высокий уровень.
  6. Установить разряды MSTR и SPE регистра SPI_CR1 для перевода модуля в режим ведущего и включения его.

Посмотрим как это выглядит на практике.

Рассмотрим пример конфигурирования модуля SPI1 для работы с картой памяти. Начнем с настройки порта:

RCC->APB2ENR |=  RCC_APB2ENR_AFIOEN;     //включить тактирование альтернативных функций        /
RCC->APB2ENR |=  RCC_APB2ENR_IOPAEN;     //включить тактирование порта А

Настраиваем выводы согласно выполняемым функциям:

GPIOA->CRL   &= ~GPIO_CRL_CNF5;     //
GPIOA->CRL   |=  GPIO_CRL_CNF5_1;   //
 
//вывод MISO: вход цифровой с подтягивающим резистором, подтяжка к плюсу
GPIOA->CRL   &= ~GPIO_CRL_MODE6;    //
GPIOA->CRL   &= ~GPIO_CRL_CNF6;     //
GPIOA->CRL   |=  GPIO_CRL_CNF6_1;   //
GPIOA->BSRR   =  GPIO_BSRR_BS6;     //
 
//вывод MOSI: выход двухтактный, альтернативная функция, 50MHz
GPIOA->CRL   |=  GPIO_CRL_MODE7;    //
GPIOA->CRL   &= ~GPIO_CRL_CNF7;     //
GPIOA->CRL   |=  GPIO_CRL_CNF7_1;   //

 

Теперь настроим модуль SPI.

1
2
3
4
5
6
7
8

RCC->APB2ENR |= RCC_APB2ENR_SPI1EN; //подать тактирование                                      /
SPI1->CR1     = 0x0000;             //очистить первый управляющий регистр
SPI1->CR2     = 0x0000;             //очистить второй управляющий регистр
SPI1->CR1    |= SPI_CR1_MSTR;       //контроллер должен быть мастером    
SPI1->CR1    |= SPI_CR1_BR;         //задаем скорость
SPI1->CR1    |= SPI_CR1_SSI;        //обеспечить высокий уровень программного NSS
SPI1->CR1    |= SPI_CR1_SSM;        //разрешить программное формирование NSS
SPI1->CR1    |= SPI_CR1_SPE;        //разрешить работу модуля SPI

Расписал все по битам, поэтому особых пояснений, думаю, не нужно.

 

Настройка модуля закончена. Чтобы запустить процесс передачи, необходимо загрузить передаваемые данные в регистр DR.

1

SPI1->DR = data;                       //загружаем данные для передачи                         /

Окончание передачи контролируем проверкой разряда TXE регистра  SPI_SR:

1

while (!(SPI1->SR & SPI_SR_TXE));      //ожидание окончания передачи                           /

Одновременно с передачей идет прием данных. Принятые данные считываем с регистра данных:

1

tmp = SPI1->DR ;                       //читаем принятые данные                                /

Чтобы проверить работоспособность модуля без ведомого можно применить одну хитрость. Соединяем выводы MISO и MOSI между собой и посылаем данные. Если все нормально, должны получить переданные данные.

    • Info:
      14.10.2012 в 21:19

      Всё тут правильно указано и достоверно работает.
      Поймите разницу TXE и RXNE:
      Изначально при инициализации SPI биты RXNE=0 TXE=1.
      TXE – устанавливается при опустошении буфера передатчика
      RXNE – устанавливается когда буфер приемника содержит данные, то есть не пустой

      Вот в исходнике мы и ждем когда очистится буфер передатчика чтоб пойти дальше по алгоритму.
      Вы предлагаете продолжать ход программы только когда в буфере приемника появятся данные и сбросится он только после того как эти данные запросят (It is cleared
      when SPI_DR is read).
      Учитывая что SPI можно настроить только на передачу данных, то какой смысл вам проверять что в приемнике и этим вы можете завести программу в бесконечный цикл ожидания что там мы не приняли еще.
      Простыми словами !(SPI1->SR & SPI_SR_RXNE) нужен для проверки наличия данных на приемнике SPI. НО это лучше на прерываниях делать или при помощи DMA иначе ошибок в программе не оберетесь…
